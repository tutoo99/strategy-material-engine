# Buildmate 阶段一 Fast Path

这份文档只服务一个目标：**在长上下文模型环境里，用最少的 LLM 轮次跑完阶段一。**

如果当前任务是：

- 拆一篇原贴
- 把一份整理好的病历样入库
- 跑阶段一注册流程

先读这份文档。默认不要再读 `workflow.md`、`quality-bar.md`、大段脚本源码。

## 目标

阶段一成功路径优先压到：

- `2~3` 次主要推理回合
- `0` 次无必要长文档补读
- `0` 次空转等待用户催
- `0` 次中途分析型回合

## 成功路径与失败路径

### 成功路径

成功路径只允许做 4 件事：

1. 输入判别
2. 生成结构化稿
3. 跑 `run_stage1_pipeline.py`
4. 交付摘要

### 失败路径

只有出现真实失败，才允许升级：

1. 先看错误文本
2. 如果是白名单格式错误，先跑 `auto_repair_case.py`
3. 如果仍失败，再定点补读一个相关文件
4. 不允许一上来扫目录、读全 schema、看全脚本源码

## 最小加载集

阶段一默认只加载：

1. 这份 `stage1-fast-path.md`
2. 用户给的原贴或整理稿

只在下面两种情况补读其他文档：

- 缺少必要字段，需要看管理员模板
- 成功路径失败，需要补 `assets/cases/` 结构或定位报错

此时才允许补读：

- `admin-operator-manual.md`
- `schema.md` 里 `assets/cases/` 相关部分

## 输入判别

只分两类：

- `【输入类型】原贴`
- `【输入类型】整理好的病历样`

如果用户没声明，先追问这一个字段；补齐后必须自动继续。不要自己猜。

再做一个快速判别：它是不是单案例材料。

优先进入阶段一的材料：

- 单篇个人经验帖
- 单篇复盘帖
- 单篇访谈整理
- 已按六步法整理好的单案例稿

不要直接把下面这些当成单 case 跑：

- 综合操作手册
- 官方运营指南
- 多案例拼盘
- FAQ 型说明书

## 固定执行序列

阶段一默认固定为这 4 步：

### 阶段一（1/4）输入判别

- 确认 `原贴 / 整理好的病历样`
- 确认是不是可拆的单案例
- 如果字段已齐，立刻进入下一步

### 阶段一（2/4）生成结构化稿

- 如果是 `原贴`，第一动作不是直接写结论，而是先搭一个朴素文字案板
- 案板必须在简单文档工具里建立，允许：
  - 飞书文档
  - Word
  - Notion
  - 手机备忘录
- 案板最顶部必须先写：
  - `帖子标题`
  - `原文链接`
- 然后再往下搭最朴素的文字框架；不要一上来做复杂表格、数据库或多维表
- 推荐直接使用 `references/stage1-workbench-template.md`
- 搭完案板后，先完整通读原贴一遍；这一遍禁止做笔记、划线、边读边拆字段
- 第一遍读完后，只允许先写 `一句话业务`
- `作者是谁 / 启动资源 / 核心目标 / 最终结果` 必须等到第二遍精读再补
- 第二遍精读时，只允许像扫描仪一样抓四类信息：
  - `作者是谁`
  - `启动资源`
  - `核心目标`
  - `最终结果`
- 第二遍精读优先保留原文中的数字、周期和量化结果，不要先改写成模糊表述
- 第二遍结束时，`【案例身份证】` 必须补满
- 第三遍细读时，只允许抓“十字路口”和“动作”
- 重点识别作者做选择的信号句：
  - `我决定...`
  - `我选择了...`
  - `在 A 和 B 之间，我选了...`
  - `试了 X 不行，于是改成 Y`
- 每个 `决策点` 必须补齐：
  - `决策点`
  - `选择`
  - `依据`
  - `动作`
- `依据` 必须写清楚为什么这么选；没有“为什么”的决策点视为不完整
- `动作` 必须尽量细到工具、入口、关键词、按钮、参数
- 原文没写清楚具体动作时，必须显式标成 `【推测】`
- 记录动作时，默认同时补一条 `作战序列`，说明它在整条路径里排第几步
- 提炼心法前，先关掉原帖，只允许看 `【案例身份证】` 和 `【决策地图】`
- `作战三原则` 必须写成可检查的动作纪律，不能写成空泛口号
- `最大一个坑` 必须同时给出作者验证过的解决方案
- `最值钱一句忠告` 必须能直接改变新手的一个执行动作
- 额外补一块 `归因与边界`，把“这更像因果还是相关性”写清楚
- 原文里出现链接、模板、工具入口时，补进 `资源清单`
- 第六步必须执行最终检查：
  - 检查 `案例身份证 -> 决策地图 -> 核心心法` 是否逻辑自洽
  - 把每个 `动作` 大声读一遍，确认新手听完就能开始做
  - 如果动作听完仍不知道第一步是什么，必须回写重修
- 最终在文档最前面补 `【案例名称】`
- 命名格式固定为 `作者身份 + 核心打法 + 结果`
- 如果是 `原贴`：一次性完成六步拆解正文，只落文件，不在对话里刷长稿
- 如果是 `整理好的病历样`：快速检查是否已有 `案例身份证 / 决策地图 / 核心心法`
- 不要在这一步读 schema 全文
- 不要在这一步中途复盘耗时

### 阶段一（3/4）运行 pipeline

- 原贴默认跑：

```bash
/opt/miniconda3/bin/python3 scripts/run_stage1_pipeline.py \
  <source-file> \
  --root . \
  --input-type source
```

- 整理好的病历样默认跑：

```bash
/opt/miniconda3/bin/python3 scripts/run_stage1_pipeline.py \
  <case-file> \
  --root . \
  --input-type case
```

- 只有输入稿已经通过 preflight，才单独跑：

```bash
/opt/miniconda3/bin/python3 scripts/register_case.py \
  <case-file> \
  --root .
```

### 阶段一（4/4）交付摘要

- 默认只交付文件路径、简短摘要、校验结果、索引结果
- 默认不展开完整拆解稿

## 最小 case 模板

成功路径默认不必去读 `schema.md`，直接按下面最小结构生成即可。

如果当前输入是 `原贴`，建议先用下面这个“案板起稿顺序”再生成正式 case：

```text
文档顶部：
- 帖子标题：...
- 原文链接：...

然后继续：
- 【案例身份证】
- 【决策地图】
- 【核心心法】
```

```yaml
---
title: <案例名称>
author: <作者是谁>
date: <YYYY-MM-DD 或 unknown>
source_type: post
source_title: <帖子标题>
source_url: <原文链接>
platform: <来源平台>
topic: <主题>
verification: verified | unverified | unknown
source_path: <sources/...>
quality_score: 2.5
status: draft
content_source: ai_generated | human_provided | human_approved
---
```

```markdown
## 【案例身份证】
**帖子标题：**
**原文链接：**
**一句话业务：**
**作者是谁：**
**启动资源：**
**核心目标：**
**最终结果：**

## 【决策地图】
**1. 决策点：**
- **选择：**
- **依据：**
- **动作：**
  - **工具：**
  - **操作：**
  - **参数：**
- **是否推测：** 是/否
- **证据摘录：**

## 【核心心法】
### 作战三原则：
1. ...
2. ...
3. ...

**最大一个坑：**
**最值钱一句忠告：**

## 【归因与边界】
**成功归因判断：**
**交叉验证案例：**
**反事实追问：**
**平台/时间上下文：**
**账号/业务阶段：**
**适用边界：**

## 【作战序列】
1. ...

## 【资源清单】
**资源链接：**
**资源状态：**
**最后检查：**

## 【待验证推测】
- ...
```

关键约束：

- `是否推测` 只能写纯 `是/否`
- 原文没写清楚的动作，必须显式标注为推测
- 原贴必须按六步法生成正文；不要跳过“检查与命名”

## 成功路径禁止事项

在阶段一没有失败前，不要做这些动作：

- 不要中途复盘耗时
- 不要中途长篇解释 schema
- 不要反复搜索 `scripts/`
- 不要为了“保险”把阶段二文档全读一遍
- 不要把完整拆解稿刷进对话上下文
- 不要在 `2/4 -> 4/4` 之间等待用户再说一次“继续”
- 不要出现“先汇报一下进度，后面再做”的空转回合

## 默认交付模板

阶段一跑完后，默认只交付：

```text
当前阶段：阶段一（完成）
当前动作：交付阶段一结果摘要，不默认展开完整拆解稿
交付方式：文件路径 + 简短摘要 + 校验/入库结果
输入类型：source / case
草稿文件：...
案例文件：...
校验结果：通过 / 未通过
自动修复次数：N
索引结果：...
下一步：如需查看全文，再打开对应文件；默认不在对话里粘贴整稿
```

只有用户明确说：

- `把完整拆解稿贴出来`
- `不要摘要，直接展开全文`

才允许展开正文。

## 进度汇报协议

阶段一至少播报这 4 个节点：

- `阶段一（1/4）输入判别`
- `阶段一（2/4）生成结构化稿`
- `阶段一（3/4）运行 pipeline`
- `阶段一（4/4）交付摘要`

如果同一节点超过 `3` 分钟，必须补一次：

- 当前还停留在哪一步
- 已完成什么
- 还差什么
- 预计剩余时间

但记住两条硬规则：

1. **进度汇报不能替代执行动作**
2. **进度汇报必须和下一执行动作同回合出现**
