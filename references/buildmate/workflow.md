# Buildmate 阶段一 / 阶段二 / 阶段三 / 阶段四工作流

说明：

- 如果当前任务是**阶段一**，且目标是缩短长上下文模型的推理时间，**不要先通读本文件**
- 先读 `references/buildmate/stage1-fast-path.md`
- 只有阶段一成功路径真实失败后，才回到本文件补完整规则
- 本文件更适合作为完整规则总表和补充参考，而不是阶段一成功路径的默认入口

## 目标

把案例从原始帖子沉淀成“可复用基因”，再把基因库训练成：

- 能解决具体商业症状的阶段二专家模型
- 能围绕明确目标组装方案包的阶段三策略路由器
- 能围绕真实执行反馈持续更新用户状态与下周任务包的阶段四智能体层

## 核心原则

- 阶段一优先遵守 **“第一阶段六步拆解法”**
- 阶段二优先遵守 **“用户给症状 → 多轮澄清 → 病例会诊 → 诊断手册 → 治疗包 → 临床实习”** 的固定链路
- 脚本的作用是辅助起草、注册、校验、索引；**不能替代人工阅读、提炼和诊断判断**

## 试运行期工作流

阶段一、阶段二试运行时，建议严格按下面这条链路跑：

### A. 阶段一建库

1. 系统管理员先声明输入类型：`原贴` 或 `整理好的病历样`
2. 如果是 `原贴`：
   - 先进入 `sources/`
   - 再按六步拆解法整理
   - 通过 `run_stage1_pipeline.py` 自动完成校验、自修复、注册、索引
3. 如果是 `整理好的病历样`：
   - 先判断它是否已经满足“可直接入库”
   - 满足则直接进入 `run_stage1_pipeline.py --input-type case`
   - 不满足则先补齐六步结构，再进入自动流水线
4. 字段补齐且开工后，阶段一默认是**自动串行流程**，不得在 `质量校验 -> 注册入库 -> 构建索引` 中间等待用户再次确认
5. 只有缺少必要字段或出现不可自动修复错误时，才允许中断问用户

### A.1 阶段一交互顺序

系统在阶段一必须按下面顺序主动互动：

1. 播报：`当前阶段：阶段一（1/4）输入判别`
2. 发编号选择：
   - `1. 原贴`
   - `2. 整理好的病历样`
3. 根据编号发对应模板
4. 若字段缺失，逐项追问
5. 字段补齐后，播报：`当前阶段：阶段一（2/4）六步拆解`
6. 拆解完成后，播报：`当前阶段：阶段一（3/4）质量校验`
7. 如果校验只命中自动修复白名单，系统必须先自动修复再复检，不得要求用户手动 patch
8. 通过校验后，播报：`当前阶段：阶段一（4/4）注册入库`
9. 注册完成后必须自动构建索引，并交付文件路径与摘要
10. 进度汇报不能替代执行动作；只汇报不继续执行视为协议违例
11. 阶段一默认交付不能直接粘贴完整长稿，必须先走固定摘要模板
12. 如果阶段一任一步骤持续超过 `3` 分钟，必须追加一次进度汇报，并说明：
   - 当前仍停留在哪一步
   - 已完成到哪里
   - 还缺什么
   - `预计剩余时间`

### B. 阶段二问诊

1. 用户提出一个具体痛点问题
2. 系统先多轮澄清，收敛成标准靶子
3. 从阶段一基因库召回正式病例
4. 组装检查站点与行动工单
5. 运行 `validate_session.py`
6. 运行 `finalize_session.py`
7. 交付最终工单给用户执行

### B.1 阶段二交互顺序

系统在阶段二必须按下面顺序主动互动：

1. 播报：`当前阶段：阶段二（1/6）问题收集`
2. 主动要求用户用一句话描述当前最具体的痛点
3. 若问题不够具体，播报：`当前阶段：阶段二（2/6）靶子澄清`
4. 发填空模板：
   - `1. 【平台】`
   - `2. 【卡住的环节】`
   - `3. 【可观察现象】`
   - `4. 【目标结果】`
5. 收敛后，必须回显“澄清后的标准靶子”
6. 再播报：
   - `阶段二（3/6）病例召回`
   - `阶段二（4/6）工单组装`
   - `阶段二（5/6）校验收口`
   - `阶段二（6/6）交付执行`
7. 交付工单后，明确告诉用户当前已进入执行期，等待反馈
8. 如果病例召回、工单组装或校验收口持续超过 `3` 分钟，必须追加一次超时进度汇报

### C. 阶段二反馈

1. 用户按工单执行
2. 用户按固定反馈格式回传结果
3. 系统把反馈落到对应 `session` 文件中的 `用户反馈`
4. 系统触发 `self_repair`
5. 如果反馈证明工单无效或部分有效：
   - 修正阶段二工单
   - 必要时触发系统补库
6. 如果反馈证明该样本已经商业化跑通并赚到钱：
   - 不直接替代原阶段一病例
   - 但可以生成“阶段一补录 / 回写候选”动作

### C.1 阶段二反馈交互顺序

系统在反馈入口必须按下面顺序主动互动：

1. 播报：`当前阶段：阶段二反馈（1/3）反馈收集`
2. 主动发出固定反馈模板
3. 如果用户只给一句自然语言感受，继续追问缺失字段
4. 满足最小反馈条件后，播报：`当前阶段：阶段二反馈（2/3）自修正`
5. 完成更新后，播报：`当前阶段：阶段二反馈（3/3）更新完成`
6. 如果自修正持续超过 `3` 分钟，必须追加一次进度汇报

## 编号选择与填空规则

- 需要用户做分支判断时，统一优先给编号选项
- 每组编号选项控制在 `2~4` 个
- 需要用户补信息时，统一优先给填空模板
- 不允许把“你再详细说说”当作默认交互动作
- 系统每次追问都必须说明：
  - 现在在哪一步
  - 还缺什么字段
  - 补完后会进入哪一步
- 试运行期间，系统每次关键播报还必须说明：
  - 正在做什么
  - 预计多久结束

### 整理好的病历样输入协议

```text

【案例名称】...
【是否已商业化验证】是 / 否
【是否已赚到钱】是 / 否

- 【决策地图】
- 【核心心法】

```

### 强制判别规则

- 看见 `【输入类型】原贴`，按原贴处理
- 看见 `【输入类型】整理好的病历样`，按整理稿处理
- 看不见 `【输入类型】`，必须先追问
- 不允许系统自己猜“这大概是原贴还是病历样”

## 第一阶段六步拆解法

### 1. 准备工作台

- 先打开一个最顺手的朴素文档工具，再开始拆帖子
  - 飞书文档
  - Word
  - Notion
  - 备忘录
- 在文档最顶部先写：
  - `帖子标题`
  - `原文链接`
- 再往下手动搭一个最朴素的文字案板；不要一上来先做复杂表格、数据库或多维表
- 模板至少包含：
  - `案例身份证`
  - `决策地图`
  - `核心心法`
- 推荐直接使用固定案板框架：
  - `【案例身份证】`
  - `帖子标题：`
  - `原文链接：`
  - `一句话业务：`
  - `作者是谁：`
  - `启动资源：`
  - `核心目标：`
  - `最终结果：`
  - `【决策地图】`
  - `1. 决策点：`
  - `选择：`
  - `依据：`
  - `动作：`
  - `【核心心法】`
  - `作战三原则：`
  - `最大一个坑：`
  - `最值钱一句忠告：`
- 这是“搭案板”，不是“直接下结论”

### 2. 第一遍通读

- 现在打开你已经选定的那篇精华帖，从头到尾先完整读一遍
- 这一遍要像读故事一样读；不要做任何笔记，不要划线，也不要一边读一边拆字段
- 这一轮唯一任务是先看懂：这个人最开始是什么情况，后来想干嘛，最后做成了什么
- 读完后，先在脑子里用一句大白话复述这个主干；不用先写长分析
- 然后回到 `【案例身份证】`，第一轮只填写 `一句话业务`
- 除 `一句话业务` 外，其他字段都不要在这一轮提前填写

### 3. 第二遍精读

- 现在开始第二遍精读；这一遍眼睛要像扫描仪，只找四类具体信息
- 先找 `作者是谁`
  - 优先找作者在开头或结尾对自己身份和背景的介绍
  - 重点抓：他是什么身份、之前做什么、处在什么生活或职业阶段
- 再找 `启动资源`
  - 重点抓：启动资金、投入时间、是否单人、是否有合伙人、用到什么技能或设备
- 再找 `核心目标`
  - 重点抓：他最初最想达成的那个具体目标，不要泛化成空话
- 最后找 `最终结果`
  - 重点抓：赚了多少钱、涨了多少粉、用了多久、拿到了什么明确结果
- 数字、周期、量化信息优先保留原文表达；能抄原句就不要自行改写成模糊总结
- 这一轮结束时，`【案例身份证】` 应该补满：
  - `作者是谁`
  - `启动资源`
  - `核心目标`
  - `最终结果`

### 4. 第三遍细读

- 现在开始第三遍细读；这一遍要像侦探一样，专门找作者做过“选择”的地方
- 重点留意这些信号词：
  - `我决定...`
  - `我选择了...`
  - `当时在 A 和 B 之间，我选了...`
  - `我试了 X，但发现不行，于是改成了 Y...`
  - `最重要的一个动作是...`
- 每找到一个，就在 `【决策地图】` 下新建一个 `决策点`
- 每个 `决策点` 必须包含：
  - `决策点`
  - `选择`
  - `依据`
  - `动作`
- `决策点` 要写成他当时面临的那个十字路口，而不是笼统地复述结果
- `依据` 是最关键字段，必须回到上下文找出他为什么这么选
- `动作` 不能只写“做了内容 / 做了投放 / 做了优化”这类模糊话，必须尽量细到：
  - 打开什么工具
  - 进入什么页面或入口
  - 搜索什么关键词
  - 点击什么按钮
  - 改哪些参数
- 如果原文没有写清楚具体动作，必须诚实写成：
  - `【原文未提供具体动作，推测可能是：...】`
- 原文没有的部分，一律显式标 `【推测】`

### 5. 提炼心法

- 到这一步先关上原帖，不要再回去看
- 脱离原文，只看已经整理好的 `【案例身份证】` 和 `【决策地图】`
- 提炼：
  - `作战三原则`
  - `最大一个坑`
  - `最值钱一句忠告`
- `作战三原则` 不能写成“坚持 / 专注 / 别放弃”这类空话，必须写成可检查的动作纪律
- `最大一个坑` 不能只写问题本身，必须连同作者验证过的解决方案一起写出
- `最值钱一句忠告` 必须能让一个新手立刻改变一个执行动作

### 6. 检查与命名

- 最后从头到尾再看一遍整份拆解报告
- 先检查逻辑是否自洽：
  - `【案例身份证】` 里的资源条件
  - `【决策地图】` 里的选择与动作
  - `【核心心法】` 里的原则与忠告
  - 三者之间必须能串起来，不能互相打架
- 再检查 `动作` 是否真的可执行：
  - 把每一个 `动作` 项大声读出来
  - 如果一个完全没做过的新手听完后就知道第一步该做什么，说明动作合格
  - 如果听完仍然不知道怎么开始，就回去把动作重写到更具体
- 最后在文档最前面补一行 `【案例名称】`
- 案例名称必须按 `作者身份 + 核心打法 + 结果` 命名

## 系统流程

### 1. 收原料

- 把原始案例放入 `sources/`
- 补齐标题、链接、平台、领域、日期、标签
- 如果原文很长，优先保留完整原文，再加你的摘录

### 2. 起草结构化案例

- 跑 `scripts/extract_case.py`
- 生成 `assets/case_drafts/` 里的草稿
- 不要直接信任草稿；它只是帮你减少“从零开写”的成本
- 草稿产出后，仍应回到“六步拆解法”补齐正文

### 3. 人工补全

重点补四类最影响后续复用质量的信息：

- `作者是谁`
- `启动资源`
- `决策地图`
- `动作步骤`
- `核心心法`

特别是 `动作步骤`，要尽量写到：

- 打开什么工具
- 搜索什么关键词
- 改哪些参数
- 在什么时机发布/执行
- 如果原文没写清，必须显式改写成 `【推测】...`

### 4. 人工确认并冻结正文

如果这版内容已经被你认可，后续目标就不是“让系统再整理一遍”，而是“无损沉淀”：

- 认可前：它是 `draft`，允许重生成
- 认可后：它应进入 `assets/cases/`，并标记为 `approved + body_lock: true`
- 一旦正文锁定，后续机器只能补标签、索引、资源引用，不能删减或压缩正文

如果你是直接给一份已经整理好的 Markdown：

- 不走 `extract_case.py` 重写正文
- 直接进入 `scripts/run_stage1_pipeline.py --input-type case`
- 前提是这份 Markdown 已经过“六步拆解法”检查

### 5. 质量校验

- 跑 `scripts/validate_case.py`
- 如果有错误，先判断是否属于自动修复白名单
- 属于白名单时，必须先跑 `scripts/auto_repair_case.py` 自动修复并复检
- 不属于白名单时，才中断并要求补齐具体缺项
- 如果有警告，优先修动作不具体、原则太空泛、推测未标记这三类问题

### 6. 自动流水线

默认优先跑：

```bash
python3 scripts/run_stage1_pipeline.py <input-file> --root .
```

这个脚本负责连续执行：

- 输入判别
- 原贴抽取草稿或病历样直接校验
- `validate_case.py`
- `auto_repair_case.py`
- `register_case.py`
- `build_case_index.py`

如果脚本中断，必须输出：

- 哪个阶段中断
- 哪些错误不可自动修复
- 用户需要补齐哪些字段

### 6.1 阶段一默认交付模板

阶段一自动流水线跑完后，默认先交付：

```text
当前阶段：阶段一（完成）
当前动作：交付阶段一结果摘要，不默认展开完整拆解稿
交付方式：文件路径 + 简短摘要 + 校验/入库结果
草稿文件：...
案例文件：...
校验结果：通过 / 未通过
自动修复次数：N
索引结果：...
下一步：如需查看全文，再打开对应文件；默认不在对话里粘贴整稿
```

只有当用户明确要求“把完整拆解稿贴在对话里”时，才允许展开全文。

### 7. 索引入库与检索

- 自动流水线会默认构建 JSONL 索引
- 手动重建时跑 `scripts/build_case_index.py`
- 后续检索统一走 `search_cases.py`

适合这些查询：

- “找 3 个小红书冷启动案例”
- “找使用稿定设计做封面的案例”
- “找低成本起盘、3 个月内有变现结果的案例”
- “找跟私域破冰相关的历史案例”

## 推荐节奏

- 先做 5~10 个高质量案例，不要一上来堆数量
- 每个案例至少补出 3 个有效决策点
- 每周抽一次样，修正你最常见的抽取错误
- 草稿和最终版分开存；不要让 AI 草稿覆盖人工确认稿

## 与后续阶段的衔接

- 阶段二会按 `symptoms` 召回相关病例，并产出诊断手册与治疗包
- 阶段三会按 `strategy_tags` 和 `resource_refs` 做节点连边
- 阶段四会结合 `result_summary`、`quality_score`、`status` 选择优先调用的案例

## 阶段二固定流程

### 目标

把阶段一的“死数据”先变成一个明确的专家训练靶子，再送进专家熔炉，最终围绕具体症状输出一份可直接执行的工单。

阶段二有两种模式：

- **建模模式**：从案例基因库中选择第一个最高频、最具体、最可诊断的商业症状，并创建 `【专家模型训练场：XXX诊断】`。
- **运行问诊模式**：用户带着当前症状进入专家熔炉，基于训练场、病例和运行时召回生成诊断工单。

### 0. 先决条件

- 已有可检索的阶段一基因库
- 案例里已经尽量补好 `symptoms`
- 当前目标是做“诊断检查单 + 治疗包”，不是做万能策略顾问
- 阶段二使用的“病历样本”，只认阶段一入口中**已经商业化验证**的病例
- 建模模式必须可以写入 `expert_models/training_grounds/`
- 运行问诊模式继续使用统一 `expert_models/furnace.md` 和 `expert_models/sessions/`

### 1. 选定第一个靶子

阶段二建模模式必须从阶段一基因库开始，而不是从系统主观猜测开始。

固定动作：

1. 打开阶段一案例基因库：优先读取 `index/cases/cases_meta.jsonl`，必要时回到 `assets/cases/` 正文。
2. 扫描每个案例的 `symptoms`、结果摘要、决策地图和搜索正文。
3. 提取“大家最常喊疼、最反复出现、最具体”的候选症状。
4. 明确排除大而空的问题，例如：
   - “怎么赚钱”
   - “怎么引流”
   - “怎么做增长”
   - “怎么做内容”
5. 保留可诊断靶子，例如：
   - “小红书笔记小眼睛卡在 200-300，上不去”
   - “抖音视频完播率低”
   - “私域加了好友不说话，无法破冰”
   - “独立站点击率高，但转化率极低”
6. 按 `出现频次 -> 具体度 -> 可观察指标 -> 可召回病例数 -> 商业化验证强度` 排序，选出第一靶子。

本步骤必须产出：

- `index/stage2/symptom_candidates.jsonl`：高频症状候选表
- `index/stage2/target_selection_report.md`：靶子选择报告
- `expert_models/training_grounds/<slug>.md`：标题为 `【专家模型训练场：XXX诊断】` 的训练场页面

如果当前基因库没有任何合格具体症状，不得强行建模；必须输出“阶段一补库工单”，由系统负责补库。

### 2. 创建专家模型训练场

训练场页面必须先完成“靶子定义”，再进入会诊。

必填字段：

- `训练场标题`：必须使用 `【专家模型训练场：XXX诊断】`
- `第一靶子`：一句话写清楚具体症状
- `排除项`：明确说明本训练场不解决哪些泛问题
- `出现频次`：来自阶段一基因库统计
- `代表案例`：列出命中的阶段一案例路径
- `可观察指标`：例如阅读量、完播率、转化率、回复率、出单数
- `适用平台 / 场景 / 环节`
- `证据状态`：`formal / bootstrap / provisional`
- `下一步会诊关键词`：用于第二步召集会诊病例

训练场不是抽象概念页，而是后续会诊、诊断手册、问诊表单和治疗包的工作台。

### 3. 运行问诊入口：用户提出症状

运行问诊模式先让用户描述当前最具体、最反复出现的商业问题。

合法靶子来源只有四类：

- 用户当前明确提出的具体症状
- 用户授权系统从指定案例中提炼出的具体症状
- 阶段四红色警报自动触发的具体症状
- 阶段二建模模式已经选出的第一靶子

不允许系统凭空预设一个默认靶子强行开始。

### 4. 多轮澄清并收敛靶子

必须连续追问，直到问题满足以下四点：

- 是具体症状，不是大问题
- 有可观察现象或指标
- 有明确平台 / 场景 / 环节
- 能在当前基因库里召回到足够多的相关病例

如果不满足，就继续追问；**未收敛前，不进入阶段二产物编写**。

### 5. 熔炉内部召回病例

- 通过 `symptoms`、正文检索、关键词筛选等方式，召回 `5~10` 个相关病例
- 这些病例必须来自 `assets/cases/`，且应已具备市场验证 / 商业化落地结果
- 外部公开视频、transcript、教材、行业分析文，只能作为辅助参考，不能计入正式病例数
- 如果正式病例数 `<5`：
  - 本次只允许作为 `bootstrap / provisional` 会话运行
  - 可以产出工单，但必须显式写出证据不足，不得声称是成熟专家结论
  - 会话中必须记录 `evidence_case_count` 与 `evidence_gap`
- 如果某条检查逻辑缺少病例支撑：
  - 先移除该检查逻辑
  - 禁止用经验脑补成确定步骤
  - 记录到 `self_repair` 中
- 如果正式病例数为 `0`：
  - 不输出伪装成可执行的业务诊断工单
  - 只输出“阶段一补库工单”
  - 补库执行方默认是系统，不是用户
- 对每个病例记录：
  - 症状描述
  - 最终结果
  - 关键决策 / 动作
  - 医生诊断结论
  - 引用案例路径

### 6. 熔炉内部组装 SOP

- 将会诊结果整理成“检查站点”
- 每个站点至少写出：
  - 检查项
  - 判断方法
  - 行动指令
  - 参数
  - 资源编号 / 资源引用
  - 对应病例编号
- `资源编号` 是硬约束，不是可选项。每个检查站点都必须至少绑定 `1` 个资源编号，例如 `R-01`
- `资源编号` 必须能回链到阶段一病例中的具体资源，而不是空泛写“参考同行”
- `行动指令` 必须落到操作台级别，至少写出：
  - 动作：打开什么工具 / 点击什么入口 / 搜索什么词
  - 参数：字号 / 颜色 / 时间 / 阈值 / 标题结构等
  - 资源编号：当前动作使用哪个资源包
  - 资源内容：模板名 / 工具链接 / 配色参数 / SOP 名称
  - 来源病例：这个资源是从哪条病例里抽出来的
- 如果某条检查站点暂时找不到资源编号，就不能进入正式诊断手册；必须回到病例库补齐资源映射

### 7. 输出工单 / 治疗包

- 问诊表单负责采集症状
- 问诊表单前半部分继续负责采集症状，后半部分必须产品化输出“诊断报告”
- 诊断报告不能是几段分析文字，必须是：
  - 待办事项清单
  - 治疗包
  - 工单
- 治疗包负责输出按顺序执行的动作清单
- 结果必须像工单：
  - 任务名
  - 动作
  - 参数
  - 参考案例
  - SOP / 资源链接
- 推荐使用固定报告结构：
  - `## 您的专属优化方案`
  - `**诊断结论**：...`
  - `**✅ 请按顺序执行以下动作（预计总耗时：XX分钟）**`
  - `### 任务一：...`
  - `### 任务二：...`
  - `### 任务三：...`
- 每个任务至少包含：
  - 动作
  - 参数
  - 参考案例
  - SOP / 资源
  - 预计耗时
- 用户拿到结果后，不应该还要二次思考“接下来怎么做”；默认应能直接按任务顺序操作
- 同时输出两种格式：
  - Markdown 工单：给人直接执行
  - YAML 工单：给系统直接消费
- Markdown 与 YAML 中的任务顺序、任务标题和核心动作必须一一对应

### 8. 运行时会话沉淀

- 每次真实诊断只保留一个运行时会话文件
- 文件中顺序记录：
  - 用户输入
  - 澄清后的标准靶子
  - 召回到的病例
  - 证据状态（`formal / bootstrap`）
  - 熔炉内部组装出的检查逻辑
  - 自修正动作
  - 阶段一补库动作（如触发）
  - Markdown 工单
  - YAML 工单

### 6.1 会话校验与自修正

- 生成会话后，必须运行 `scripts/validate_session.py`
- 如果发现违反证据规则：
  - 先执行 `self_repair`
  - 删除无病例支撑的检查站点 / 任务
  - 如果正式病例为 `0`，改为系统补库模式
- 在自修正完成前，不允许把会话当成最终交付物

### 6.2 最终交付收口

- 在交付用户前，必须运行 `scripts/finalize_session.py`
- 这个脚本负责：
  - 同步 frontmatter 与 YAML `work_order`
  - 删除无病例支撑的检查站点 / Markdown 任务 / YAML 任务
  - 在正式病例不足时强制开启 `self_repair` 与 `stage1_replenishment`
  - 二次调用 `validate_session.py`
- 只有二次校验通过后，才允许将会话状态标记为 `ready`

### 7. 临床实习与反馈回流

- 临床实习不是口头测试，必须形成正式记录
- 默认创建：`expert_models/practicums/<symptom-slug>.md`
- 找真实用户或真实内容做内部测试
- 内部测试最小样本量：
  - `3` 个真实测试对象
  - 或 `3` 条真实内容样本
- 每个测试对象都必须回答固定三问：
  - 表单问题看得懂吗？
  - 诊断结论你觉得说到点上了吗？
  - 行动建议你觉得能直接操作吗？
- 每个测试对象都必须补记录：
  - 测试前原始数据
  - 按工单执行了哪些任务
  - `1~2` 天后的结果变化
  - 是否改善
- 至少收集三类反馈：
  - 表单是否看得懂
  - 诊断是否说到点上
  - 行动建议是否可执行
- 有效：回写为新病例候选
- 但只有在该样本已跑通商业化落地、证明至少赚到过钱时，才允许正式回写到阶段一 `assets/cases/`
- 如果满足“有效 + 已赚到钱”，必须在临床实习记录里生成 `阶段一回写候选` 区块，明确：
  - 候选对象
  - 候选原因
  - 下一步按阶段一入口注册
- 无效：回查诊断手册缺漏或误判点
- 部分有效：拆分“有效动作”和“无效动作”，不要把整条模型都视为正确
- 每次临床实习结束后，必须至少执行 1 项模型迭代动作：
  - 简化表单问题描述
  - 补充诊断手册检查项
  - 修正行动工单
  - 生成阶段一补录 / 回写候选

### 7.0 临床实习正式记录结构

临床实习记录至少包含：

- 测试对象 1 / 2 / 3
- 三个固定问题的回答
- 测试前原始数据
- 执行任务编号
- `1~2` 天后的结果变化
- 有效 / 无效 / 部分有效判断
- 迭代决策

如果没有 `3` 个测试对象，必须明确写出当前只完成了几份测试，以及剩余缺口。

### 7.1 用户反馈如何触发自修正

用户执行工单后，如果想触发系统自修正，必须尽量按下面格式回传：

```text
【反馈类型】阶段二执行反馈
【会话文件】expert_models/sessions/xxxx.md
【原始痛点】...
【执行任务】T1, T2, T3
【未执行任务】T4
【结果变化】...
【是否改善】是 / 否 / 部分
【是否赚到钱】是 / 否
【金额】...
【新暴露出的卡点】...
【我建议修正的地方】...
【证据】数据 / 文本 / 截图说明
```

最小可触发字段为：

- `【会话文件】`
- `【执行任务】`
- `【结果变化】`
- `【是否改善】`

如果这四项不齐，系统不能直接开始 `self_repair`，而是必须继续追问。

### 7.2 反馈进入系统后的固定动作

一旦系统识别到：

- `【反馈类型】阶段二执行反馈`
- 且带有 `【会话文件】`

## 阶段三固定流程

### 目标

把阶段一的策略标签、资源索引和阶段三证据图谱，转成一份围绕明确目标的正式方案包。

### 地图理解入口

- 在正式跑阶段三前，先看 `strategy_models/graph_center.md`
- 这个文件负责解释阶段三中的：
  - 节点
  - 边
  - 属性
  - 当前系统里这些元素分别落在哪些文件
- `graph_center.md` 是人类理解入口；`router.md` 是系统执行入口

### 0. 先决条件

- 已有可检索的阶段一基因库
- 已生成 `index/stage3/` 证据层产物
- `strategy_models/routes/` 与 `strategy_models/resources/` 已存在正式配置和资源资产

### 1. 用户提出目标

- 目标必须是“想达成什么”，而不是“哪里卡住了”
- 当前目标族以 `strategy_models/routes/goal_profiles.yaml` 为准
- 如果目标未命中，不直接失败，而是进入自治审计：
  - 先判断是不是阶段二目标翻译有误
  - 再判断是不是阶段三路由缺口
  - 最后判断是不是阶段一证据缺口

### 2. 收敛情境

必须收敛出：

- 目标
- 平台
- 用户类型
- 业务领域
- 资源约束

如果用户没给全，可以采用合理默认值，但必须记录在会话里。

### 3. 图谱召回

- 从 `index/stage3/` 召回：
  - 策略节点
  - 情境节点
  - 资源节点
  - 策略到资源边
  - 案例到策略边
  - 案例到资源边
  - 策略共现边
  - 目标到策略路由边
  - 情境到策略路由边
  - 策略到情境关系边
- 从 `strategy_models/routes/` 召回：
  - 目标族
  - 策略画像
- 从 `strategy_models/resources/` 召回：
  - 动作包
  - 模板资源
  - 工具调用

### 4. 路由判断

- 先按目标族给出主策略和组合策略
- 再按平台 / 用户类型 / 领域 / 资源约束筛选
- 如果情境边显式标记 `not_suitable_for`，必须把该策略从最终方案剔除，并转写到“情境禁区路径”
- 优先使用 `approved` 证据支撑的策略
- 如果关键策略只有 `draft` 证据，必须降级为 `bootstrap`
- 共现边只作为组合提示，不作为因果结论
- 路由边必须显式回答：
  - 触发条件是什么
  - 调用产出是什么
  - 是否需要给出不适用警报
  - 为什么该策略适合当前情境

### 5. 方案包组装

- 最终输出必须像“可直接执行的方案包”，而不是推荐词列表
- 每个策略节点应先被当成资源集装箱读取，再进入任务组装
- 节点属性至少应显式带出：
  - `action_refs`
  - `template_refs`
  - `tool_refs`
  - `preferred_case_refs`
- 每个任务必须绑定：
  - `strategy_ref`
  - `case_refs`
  - `resource_refs`
  - `success_check`
- 资源包必须拆成：
  - `action_refs`
  - `template_refs`
  - `tool_refs`
  - `platform_resource_refs`

### 6. 会话校验

- 生成会话后，必须运行 `scripts/validate_stage3_session.py`
- 如果触发自治审计，还必须运行 `scripts/validate_stage3_audit.py`
- 校验重点包括：
  - 结构化 package 完整性
  - 资源引用存在性
  - 案例引用存在性
  - 证据等级与置信度一致性
  - ready 会话是否覆盖完整 1/7~7/7 进度链路

### 7. 正式交付

- 交付物固定为：
  - 阶段三会话文件
  - Markdown 方案包
  - YAML `solution_package`
  - `strategy_models/productization/stage3_strategy_map.drawio`
  - `strategy_models/productization/stage3_questionnaire.yaml`
- 标准路由失败时，还必须追加：
  - 阶段三自治审计文件
  - 自动生成的阶段二反问结果
  - 自动生成的阶段一补库动作
- 交付时必须显式说明：
  - 证据级别
  - 主策略
  - 组合策略
  - 风险提示
  - 7 天执行反馈指标

## 阶段四固定流程

### 目标

把阶段三执行反馈转成用户级长期状态、学习动作和下一周任务包。

### 0. 先决条件

- 已有 `stage4_models/profile/owner_profile.md`
- 已有至少一个 `stage4_models/dashboards/*.md`
- 已有可回链的阶段三会话

### 1. 档案初始化

- 用 `run_stage4_cycle.py --mode init-profile` 创建 `owner_profile`
- 系统根目录应同时保留一份 `【我的商业档案】` 入口文档，作为人工查看入口
- 档案至少包含：
  - 资源画像
  - 目标与偏好
  - 当前主战场
  - 业务前线总表
  - 交互协议
  - 交互人格
  - 主动关怀里程碑
  - 红色警报协议
  - 数据同步协议
  - 档案维护
- `资源画像` 和 `目标与偏好` 建议按问答式填写，避免后续理解漂移
- `业务前线总表` 应列出全部正在经营或关注的前线，并能回指各自 `【数据看板】`

### 2. 数据同步

- 用 `run_stage4_cycle.py --mode sync-dashboard` 更新单个前线看板
- 每个看板至少记录：
  - 核心指标
  - 内容表现
  - 内容表现采集口径
  - 待办事项
  - 主动关怀消息
  - 红色警报协议
  - 红色警报消息
  - 手动同步机制
  - 预警状态
- 如果看板进入 `critical`：
  - 系统必须自动生成阶段二红色警报诊断会话
  - 系统必须同步生成 `stage4_models/dispatches/*.md` 自动派发记录
  - 自动派发记录里必须显式写出：
    - 阶段二自修正派发
    - 阶段一补库派发
    - 下一跳跟进条件
- 默认手动同步协议为：每天下午 `17:00`，花 `10分钟` 更新一次关键数据

### 3. 执行反馈回流

- 用 `run_stage4_cycle.py --mode process-feedback` 读取阶段三会话和执行反馈
- 系统必须自动生成：
  - `stage4_feedback_record`
  - 必要时更新 `frontline_dashboard`
  - 学习动作
  - 模型修正项
- 学习动作至少在以下几类中做判断：
  - `dashboard_update`
  - `profile_update`
  - `stage3_route_update_needed`
  - `stage2_diagnosis_update_needed`
  - `stage1_replenishment_needed`
  - `next_week_task_package`
- `模型修正项` 至少要把反馈拆成三类固定槽位：
  - `修正图谱`
  - `修正专家模型`
  - `更新资源库`
- 每条反馈都应给出 `有效性评分`，避免月度复盘时只能看主观描述

### 4. 每周战略复盘会

- 用 `run_stage4_cycle.py --mode generate-review` 生成 `weekly_strategy_review`
- 周复盘至少包含：
  - 生成依据
  - 本周核心判断
  - 交互人格输出
  - 主动关怀与警报
  - 本周任务包
  - 授权项
  - 学习动作
- `本周任务包` 必须尽量下钻到可执行层，而不是停留在抽象建议：
  - 每个任务应带 `诊断`
  - 每个任务应带 `预计耗时`
  - 每个任务应带 `执行步骤`
  - 每个任务应挂接 `动作包 / 模板资源 / 工具调用 / 关联案例`
  - 每个任务应给出 `成功检查`

### 4.1 月度模型修正会

- 用 `run_stage4_cycle.py --mode generate-model-review` 汇总多条 `stage4_feedback_record`
- 月度模型修正会至少包含：
  - 生成依据
  - 本月反馈概览
  - 模型修正清单
  - 修正执行决议
- 目标不是感慨，而是把本月同类反馈合并成正式修正工单

### 5. 阶段四校验

- 所有阶段四产物必须运行 `scripts/validate_stage4_artifact.py`
- 阶段四校验重点包括：
  - artifact_type 合法
  - 必要区块完整
  - structured payload 可解析
  - 上游引用存在
  - 学习动作与授权字段可用
  - 红色警报自动派发记录与阶段二会话引用存在

就必须视为“反馈入口触发”，并固定执行：

1. 把反馈写入该会话的 `用户反馈`
2. 重新检查：
   - 原工单哪些任务有效
   - 哪些任务无效
   - 是否出现新的卡点
3. 触发 `self_repair`
4. 如果反馈显示“无效 / 部分有效”：
   - 修正阶段二工单
   - 必要时触发系统补库
5. 如果反馈显示“已赚到钱”：
   - 生成“阶段一补录 / 回写候选”动作
   - 但不自动跳过阶段一入口要求


如果是超时进度汇报，额外补一句：

```text
当前已完成：...
仍在处理的原因：...
```
