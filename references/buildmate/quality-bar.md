# Buildmate 案例质量标准

## 必须满足

- 有清晰的 `一句话业务`
- 有可辨认的 `作者是谁`
- 有最基本的 `启动资源`
- 有至少 1 个 `决策点`
- 每个决策点都包含：
  - 决策点
  - 选择
  - 依据
  - 动作步骤
- 原文不确定的地方显式标记 `【推测】`
- `是否推测` 字段只能填纯 `是 / 否`

## 强烈建议满足

- 至少 3 个决策点
- 最终结果中包含数字、周期或量化变化
- 作战三原则可检查，而不是抽象口号
- 最大一个坑同时包含“坑点 + 解决方案”
- 最值钱忠告能直接改变后续执行者的一个动作

## 常见低质量模式

### 只汇报，不执行

坏例子：

- 阶段一已经拿到合法输入，只反复汇报“在处理”，但不继续跑校验、注册、索引

好例子：

- 阶段一一旦进入 `2/4`，就连续跑完 `validate_case.py -> auto_repair_case.py -> register_case.py -> build_case_index.py`

### 可自动修复错误却让用户手改

坏例子：

- `是否推测` 写成“工具具体名称为推测”后，系统停下来要求用户逐条 patch

好例子：

- 系统先用 `auto_repair_case.py` 归一化为 `是`，复检通过后继续入库

### 只有结构修补，没有内容门禁

坏例子：

- 只补默认字段、`source_refs` 和空白值，正文空泛的 method/data 也照样入库

好例子：

- `method/playbook` 至少包含可执行编号步骤，`data` 至少包含可量化证据，修补后仍不满足就标记为 `rejected`

### 只有感悟，没有动作

坏例子：

- “作者很懂用户需求，所以做成了”

好例子：

- “作者先在小红书搜索 20 个相关关键词，截图前 20 篇爆款封面，再按高对比度大字报模板重做封面”

### 只有结果，没有路径

坏例子：

- “3 个月赚了 1.5 万”

好例子：

- “先用图文测试需求，再把私信问题整理成 FAQ，最后把资料包做成标准交付页”

### 原文没写，硬写成事实

坏例子：

- “作者使用企业微信自动化完成分层运营”

好例子：

- “【推测】作者可能使用企业微信或表格手动分层，原文未说明具体工具”

### 原则太空泛

坏例子：

- “坚持”
- “专注”

好例子：

- “每次发布前必须用另一台手机预览封面，确认小图仍能看清前 10 个字”

## 审核建议

- `draft`：刚抽取出来，仍有明显推测和空白
- `queued_for_repair`：命中了可自动修补的结构问题，正在或等待修补
- `reviewed`：自动质量门禁已通过，主要字段已补齐，可以参与检索
- `rejected`：修补后仍不合格，或命中不可自动修补的内容问题，不进正式检索
- `approved`：人工确认正文版，`body_lock: true`，后续只能追加元数据和资源索引，不能删减正文

## 阶段二必须满足

- 建模模式先从阶段一案例基因库中找出最高频、最具体的症状，再建立训练场
- 靶子必须是具体症状，不是系统主观预设的大问题
- 在开始会诊前，已经完成多轮澄清并收敛成标准靶子
- 阶段二第一步必须产出：
  - 高频症状候选表
  - 第一靶子定义卡
  - `【专家模型训练场：XXX诊断】`
- 至少有 `5~10` 个相关病例支撑诊断手册
- 这里的相关病例必须是阶段一中已商业化验证的病例，而不是外部参考或未验证样本
- 每条关键检查标准都包含：
  - 检查项
  - 判断方法
  - 行动指令
  - 资源编号 / 资源引用
- 每条关键检查标准中的 `资源编号` 都是必填，不允许只写“参考案例”而没有编号
- 每条行动指令至少包含：
  - 动作
  - 参数
  - 资源编号
  - 资源内容 / 入口
  - 病例回链
- 问诊表单最终输出的是“治疗包 / 工单”，而不是抽象建议
- 问诊表单的“诊断报告”必须是任务清单，不允许只写几段分析
- 阶段二第五步必须产出正式“临床实习记录”
- 临床实习默认至少覆盖 `3` 个真实测试对象或 `3` 条真实内容样本
- 每个测试对象都必须回答固定三问：
  - 表单问题看得懂吗？
  - 诊断结论你觉得说到点上了吗？
  - 行动建议你觉得能直接操作吗？
- 每个测试对象都必须记录 `1~2` 天后的结果变化
- 最终输出同时包含：
  - Markdown 工单
  - YAML 工单
- 如果证据不足导致某条检查逻辑无病例支撑，系统已先完成自修正，而不是把瞎编内容直接交付

## 阶段二强烈建议满足

- 高频候选表里同时标记 `出现频次 / 具体度 / 可观察指标 / 代表案例`
- 训练场标题使用固定格式：`【专家模型训练场：XXX诊断】`
- 如果正式病例 `<5`，明确标记为 `bootstrap / provisional`，并披露证据不足
- 每个治疗任务都附带预计耗时或执行顺序
- 诊断报告建议固定使用：
  - `## 您的专属优化方案`
  - `**诊断结论**`
  - `**✅ 请按顺序执行以下动作（预计总耗时：...）**`
  - `任务一 / 任务二 / 任务三`
- 每条行动指令都能回链到至少一个具体案例
- 诊断手册中的资源编号建议使用固定格式：`R-01`、`R-02`、`R-03`
- 用户测试阶段至少收集 3 份真实反馈
- 临床实习建议把“有效动作 / 无效动作 / 下一轮唯一变量”拆开记录
- 有效与无效反馈都能回流到阶段一或阶段二产物中
- 如果正式病例为 `0`，自动切换到“系统补库模式”，而不是继续输出伪工单
- 反馈入口尽量包含 `会话文件 + 执行任务 + 结果变化 + 是否改善`
- 交互入口尽量包含 `阶段播报 + 编号选择 / 填空模板`
- 任务运行超过 `3` 分钟时，系统仍能持续输出带 ETA 的进度汇报

## 阶段二常见低质量模式

### 靶子太大

坏例子：

- “怎么赚钱”
- “怎么做增长”

好例子：

- “小红书笔记阅读量一直卡在 200~300，上不去”
- “私域加了好友不说话，怎么破冰”

### 没有先从基因库选靶子

坏例子：

- 还没看阶段一案例，就凭感觉决定做“增长诊断”
- 不做频次统计，直接挑一个自己熟悉的话题开始

好例子：

- 先从 `cases_meta` 和病例正文里统计高频症状，再选择“最常喊疼、最反复出现、最具体”的问题
- 先输出候选表，再正式建立 `【专家模型训练场：XXX诊断】`

### 只有诊断，没有治疗包

坏例子：

- “封面不行，建议优化封面”
- “诊断结论：封面吸引力不足。”后面没有任务清单
- 输出一段解释，但没有明确的动作顺序

好例子：

- “打开稿定设计，搜索‘小红书封面模板’，选‘知识干货’分类，字号调到 68，主色参考 #FF4D4D”
- “任务一：更换封面模板；任务二：优化标题前 10 字；任务三：设置发布后动作”

### 只有 Markdown，没有结构化出口

坏例子：

- 只给一份文字工单，后续系统无法直接调用

好例子：

- 同时给出 Markdown 工单和 YAML `work_order`，两者任务一一对应

### 没有资源引用

坏例子：

- “建议参考优秀同行”
- “参考《病例03》”但没有资源编号
- “去优化封面”但没有说明具体模板、参数和入口

好例子：

- “参考《病例03》的配色方案，引用当前会话工单里的资源编号 R-03”
- “打开稿定设计，搜索‘小红书封面模板’，选‘知识干货’分类；资源编号 R-01；主色 #FF4D4D；来源病例 `cases/...`”

### 有资源引用但没有资源编号

坏例子：

- `资源：稿定设计模板`
- `资源：火烧云数据榜单`

好例子：

- `资源编号：R-01；资源内容：稿定设计-知识干货模板第3款；来源病例：cases/...`
- `资源编号：R-02；资源内容：火烧云数据TOP100带货榜单；来源病例：cases/...`

### 没有澄清就直接开始

坏例子：

- 用户说“怎么做内容”，系统直接输出诊断手册

好例子：

- 系统先追问平台、指标、症状出现环节，再把问题收敛成“抖音视频完播率低”

### 用未验证样本充当病历

坏例子：

- 拿一个公开视频、一个 transcript 页面，直接注册成阶段一病例并用于强化专家模型

好例子：

- 公开视频或 transcript 只能作为辅助参考；只有已经跑通商业化落地、证明赚到钱的样本，才允许进入阶段一病例库

### 证据不足却装成熟专家

坏例子：

- 只有 `1~2` 个正式病例，就输出一份看起来像高置信最终结论的专家工单

好例子：

- 明确标注“当前仅有 `1` 个正式病例，本次为 `bootstrap / provisional` 工单，先用于内测，不作为成熟模型结论”

### 证据不足还继续瞎编

坏例子：

- 明明没有从基因库召回到相关商业病例，仍然补出一堆“检查站点”“优化动作”“推荐资源”

好例子：

- 删除无证据支撑的站点，标记 `self_repair`，并由系统自动发起阶段一补库动作

### 触发了 skill，却不先判别输入类型

坏例子：

- 用户只发来一坨文本，系统直接假定它是“可直接入库病历样”

好例子：

- 系统先要求管理员明确声明 `【输入类型】原贴` 或 `【输入类型】整理好的病历样`，未声明前不入库

### 用户反馈太散，无法触发自修正

坏例子：

- “我试了下，感觉一般。”

好例子：

- “【反馈类型】阶段二执行反馈；【会话文件】...；【执行任务】T1,T2；【结果变化】阅读量从 300 到 900；【是否改善】部分。”

### 没有临床实习记录

坏例子：

- 只说“我让朋友试了下，还行”
- 没有写明测试对象是谁、执行了哪些任务、1-2 天后数据怎么变

好例子：

- 有 3 份测试记录，每份都包含三问回答、执行任务、前后数据和是否改善
- 明确写出“有效动作 / 无效动作 / 下一轮唯一变量”

### 有反馈，但没有迭代动作

坏例子：

- 记录了“看不懂”“不准”，但没有回写表单或手册

好例子：

- 反馈“表单看不懂”后，直接简化问法
- 反馈“诊断不准”后，回到病例库补充常见场景并更新诊断手册
- 反馈“有效且赚到钱”后，生成阶段一回写候选

### 触发后不主动引导

坏例子：

- 用户说“用 strategy-material-engine 处理一下”，系统直接沉默开工，不说明当前在哪一步

好例子：

- 系统先播报“当前阶段：阶段一（1/4）输入判别”，再发编号选择

### 超过 3 分钟不汇报

坏例子：

- 系统进入病例召回后，连续 `5` 分钟没有任何状态更新

好例子：

- 系统在第 `3` 分钟主动补发：当前还在 `阶段二（3/6）病例召回`，已经完成关键词筛选，正在整理正式病例，预计还需 `约 3 分钟`

### 只说处理中，不给 ETA

坏例子：

- “处理中，请稍等”

好例子：

- `当前阶段：阶段二（4/6）工单组装；当前动作：正在整理 Markdown 工单与 YAML 工单的一致性；预计剩余时间：约 3 分钟`

### 该编号选择时却让用户自由发挥

坏例子：

- “你说说这是原贴还是整理稿吧。”

好例子：

- `请选择：1. 原贴  2. 整理好的病历样`

### 该填空时却不给字段

坏例子：

- “请把问题说清楚一点。”

好例子：

- `请补充：1.【平台】 2.【卡住的环节】 3.【可观察现象】 4.【目标结果】`

### 建模资产和运行会话混为一谈

坏例子：

- 既没有训练场，也没有候选表，所有问题都直接塞进熔炉
- 用户随口问一个新问题，就跳过靶子选择，马上宣称已经有成熟专家模型
- 把运行时 session 当成训练资产，导致后续没有可复用的训练场

好例子：

- 建模模式先沉淀 `training_ground / consultation / manual / form`
- 运行模式再用同一个专家熔炉处理用户当下问题，并把结果记录到 `sessions/`
