# 【策略图谱绘制中心】

## 初始地标

- **节点：** 这就是地图上的“地点”。在当前系统里，它可以是一个策略、一个资源、一个案例，后续也可以扩展为概念或工具。
- **边：** 这是连接两个地点的“路”。它用来描述关系，比如“案例使用了策略”“策略调用了资源”“策略经常和另一个策略组合出现”。
- **属性：** 这是地点和路的详细信息。比如平台、领域、证据等级、关联案例数、资源索引、成功检查、适用约束。

## 这张图到底是什么

- 不要把它想成一个高大上的“图谱系统”。
- 它本质上是一份可以持续扩展的 **超级联系清单**。
- 我们做的事，不是先追求复杂可视化，而是先把“有哪些东西”和“这些东西之间怎么连接”记录清楚。

## 当前系统中的边

- **案例 -> 策略边：** `index/stage3/case_strategy_edges.jsonl`

- **策略 -> 情境关系边：** `index/stage3/strategy_situation_edges.jsonl`

### 边当前表达的关系

- **案例 -> 策略：** 某个案例使用了什么打法。
- **案例 -> 资源：** 某个案例调用了什么工具、模板、动作包。
- **策略 -> 资源：** 某个策略落地时通常需要哪些资源。
- **策略 -> 策略：** 哪些策略经常一起出现，适合组合推演。
- **目标 -> 策略：** 什么目标条件下，这条策略边会被激活，以及命中后要调用什么资源。
- **情境 -> 策略：** 什么类型的用户 / 平台 / 领域更适合这条策略，以及哪些策略属于当前情境禁区。
- **策略 -> 情境：** 某个策略特别适用于哪类情境，或者明确不适用于哪类情境。

### 路由边当前最关键的属性

- `trigger_conditions`
- `applicable_params`
- `not_applicable_warning`
- `call_output`
- `reason`

## 图是怎么被用起来的

- 第一步：先从用户目标收敛出目标、平台、用户类型、领域、资源约束。
- 第二步：去节点层召回相关策略、资源、案例。
- 第三步：去边层确认这些东西之间是不是有真实证据连接。
- 第四步：根据节点属性和边属性过滤出更适合当前情境的路径。
- 第五步：把命中的节点和资源动态组装成阶段三方案包。

## 当前阶段三的最小地图理解

- **不是** 先画图，再想怎么用。
- **而是** 先把节点、边、属性沉淀成稳定索引，再让系统按目标自动检索和组装。
- 当前策略节点不是抽象概念节点，而是一个最小资源集装箱。
- 当前 `router.md` 负责“怎么走路”。
- 当前 `graph_center.md` 负责“解释地图是什么”。

## 后续扩展原则

- 当前已经正式落地情境节点与“适用于 / 不适用于”关系边。
- 节点可以继续增加概念节点和目标节点，但必须先定义属性。
- 边可以继续增加“会放大”“会冲突”等关系，但必须先定义触发条件。
- 不允许只加概念，不加属性。
- 不允许只画连线，不说明关系含义。
- 不允许把共现误当因果。
